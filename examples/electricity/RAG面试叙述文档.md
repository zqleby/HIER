# 设备手册 RAG 面试叙述文档

## 简历描述

> 针对设备手册文档的多层级、专业性强的特点，基于 ChromaDB 构建层级化知识库，设计层级索引 + 实体匹配 + 迭代查询的检索策略，较基础 RAG 检索准确率提升 10%。

---

## 一、整体数据流

```
原始 PDF 设备手册
    ↓  ① PDF 转 Markdown + 结构化切块
    ↓  ② 离线实体提取
    ↓  ③ 层级标注
    ↓  ④ 向量化（text-embedding-v3）→ 写入 ChromaDB

用户问题
    ↓  ⑤ entity_llm 实时提取实体
    ↓  ⑥ 向量检索 → 层级扩展（expand_to_section）
    ↓  ⑦ 实体混合打分重排序
    ↓  ⑧ answer_llm 生成答案 → 质量判断 → 不够则扩大 k 值重查（最多 3 轮）
    ↓  ⑨ 评估（ExactMatch / F1 / Precision / Recall / LLM）
```

---

## 二、预处理阶段

### 2.1 PDF 转 Markdown + 结构化切块

**面试话术：**

> 设备手册原始格式是 PDF，直接对 PDF 做文本提取和切块会丢失格式信息——比如标题层级、表格结构、列表缩进。所以第一步我先用 PDF 解析工具把手册转成 Markdown 格式。
>
> 为什么选 Markdown？因为 Markdown 天然保留了层级结构信息：`#` `##` `###` 对应章节层级，表格用 `|` 分隔，列表保留缩进。这比直接提纯文本多了一层"结构信号"。
>
> 然后基于 Markdown 进行切块。切块策略是按语义段落切，核心规则：
> - **以标题为边界**：每个 Markdown 标题（`##` / `###`）作为一个切分点，保证不跨章节
> - **表格和列表整体保留**：不把表格拆散，一个完整表格作为一个 chunk
> - **段落按语义聚合**：同一个标题下的连续正文合并，直到遇到下一个标题或超过 token 上限
>
> 切完后每个 chunk 自动获得层级元数据：标题行作为 `parent_title`，正文作为内容。最终存储格式是 JSONL，每行包含 `[层级]`（标题路径）、`[正文]`（Markdown 内容片段）。

**追问：切块大小怎么定的？**

> 没有用固定长度。是以 Markdown 标题为自然边界，一个 `###` 下的内容一般在 300-800 token 之间，正好是语义自洽的单元。如果某个章节内容特别长（比如大段操作步骤），再按段落二次切分，但会标记它们属于同一个 parent_title。

**追问：MD 转换中表格和图片怎么处理？**

> 表格保留 Markdown 格式，嵌入模型对 Markdown 表格的语义理解比纯文本好。图片暂时做 OCR 提取文字描述后插入到对应位置。后续可以考虑多模态 embedding。

---

### 2.2 离线实体提取

**面试话术：**

> 实体提取在切块完成后离线批量执行。用 LLM（qwen-plus-latest）遍历所有 chunks，提取三类实体：设备名、参数/属性名、状态/数值。
>
> 提取结果直接写到 chunk 的元数据字段里，和向量一起存入 ChromaDB。因为是离线预处理，设备手册又是相对静态的文档，检索时直接用，不影响响应速度。
>
> 检索时，query 进来先用 entity_llm 实时提取实体，然后和候选 chunks 的预标注实体做匹配（Jaccard 相似度或重叠率），计算成本可以忽略。

**追问：LLM 实体提取准确吗？**

> 我们设计了严格的 JSON 输出 prompt，要求只输出《设备名》《参数名》《数值》三类。在这个受限领域，准确率 95% 以上。少量遗漏不影响最终效果，因为还有向量相似度兜底。

**追问：为什么离线做而不是在线做？**

> 设备手册是静态文档，离线跑一次就行，检索时的延迟完全不受影响。在线做会每次多 200ms+ 的实体提取开销。而且离线可以做更精细的标注和校验，质量更高。

---

### 2.3 层级标注与向量库构建

**面试话术：**

> 层级标注的关键字段是 `parent_title`。这一信息在 Markdown 解析阶段自然获得——`##` 就是 `###` 的 parent。比如：
>
> ```
> ## 3.1 日常巡检          ← parent_title
> ### 3.1.1 外观检查        ← chunk 的层级标签
> ### 3.1.2 油温检测        ← chunk 的层级标签
> ```
>
> "3.1.2 油温检测"下的所有 chunks，parent_title 都是 "3.1 日常巡检"。
>
> 这个 metadata 存入 ChromaDB 后，检索时可以按 parent_title 做二次过滤——命中任一个子节点，把父节点下所有子节点的 chunks 全拉出来。
>
> 向量化使用 OpenAI 的 text-embedding-v3 模型，向量库用 ChromaDB 的持久化存储。

## 三、检索阶段（核心）

### 3.1 层级索引（解决上下文缺失）

**痛点：** 细粒度切块会丢失上下文。比如 chunk 里只有"额定电流 100A"，但用户问的是"X 型断路器额定电流"，如果不同型号的规格放在不同章节，向量检索很难区分。

**做法：**

> 我们在切块时保留了层级元数据（parent_title），检索到某个 chunk 后，通过层级阈值（0.6）判断是否需要把同章节的其他 chunks 也拉出来。相当于"命中一个点，展开一个面"，确保上下文完整。

**具体参数：**
- `hierarchy_threshold: 0.6`——相似度高于此值的 chunk 才触发展开，避免低相关度 chunk 引入噪声
- `hierarchy_meta_name: "parent_title"`——以父章节标题作为层级关键字段
- `expand_to_section: true`——展开时将同一 parent_title 下的所有 chunks 加入候选集

**关键词：** 层级元数据、expand_to_section、上下文完整性

**追问：层级扩展会不会引入太多噪声？**

> 通过 hierarchy_threshold（0.6）控制触发条件，只有高度相关的 chunk 才触发扩展。而且扩展后的候选集还要经过实体排序过滤，不会无脑全加。

---

### 3.2 实体排序（解决语义漂移）

**痛点：** 纯向量相似度在专业领域会漂移。比如问"变压器油温告警阈值"，向量可能匹配到"电容器温度参数"，语义上都是"温度相关"，但实体完全不同。

**做法：**

> 我们在检索流程中加入了一层实体提取和重排序。用 LLM 从 query 中提取关键实体（设备名、参数名、状态值等），然后对候选 chunks 做混合打分——向量相似度占 0.5，实体匹配度占 0.5。这让检索结果从"语义相似"变成了"实体相关"。

**具体参数：**
- `use_entity_ranking: true`
- `entity_weight: 0.5` / `similarity_weight: 0.5`
- 实体匹配分可以用 Jaccard 相似度或实体重叠率计算，代价极小

**混合打分公式：**

```
final_score = 0.5 × entity_match_score + 0.5 × vector_similarity
```

**关键词：** 实体提取、混合打分、从语义相似到实体相关

**追问：实体提取会不会增加延迟？**

> 用轻量 prompt，实际增加的延迟约 200ms，在可接受范围内。chunk 侧的实体是离线预标注的，检索时直接用，不增加开销。

**追问：为什么权重是 0.5/0.5？**

> 做了对比实验，0.3/0.7 或 0.7/0.3 都会偏向一侧，0.5/0.5 在召回率和精确率之间最均衡。

---

### 3.3 迭代查询（解决召回不足）

**痛点：** 固定 k 值要么太少不够用，要么太多引入噪声。复杂问题往往需要跨章节组合信息，单次检索的覆盖度难以保证。

**做法：**

> 我们做了自适应检索范围：首轮取 top-10，LLM 判断答案质量，不够就自动增加到 15、20、25，最多三轮。这相当于让系统自己判断"我找到的信息够不够回答这个问题"，不够就扩大范围。

**具体参数：**
- `retrieve_k: 10`——初始检索数量
- `max_retries: 3`——最多重试 3 轮
- `retry_increment: 5`——每轮 k 值增加 5（10 → 15 → 20 → 25）

**关键词：** 自适应、质量反馈、动态 k 值

**追问：怎么判断"质量不够"？**

> 通过 generation_qa_with_reference_protocol 的反馈信号，比如引用覆盖度、答案置信度低于阈值就触发重试。具体实现时结合了引用片段的覆盖率和 LLM 自评。

---

### 3.4 三个策略的协同关系

三个策略不是串行堆叠，而是互相配合：

1. **层级扩展**增大候选集——保证上下文完整，但也可能引入噪声
2. **实体排序**过滤噪声——在扩大的候选集上精排，把正确的排到前面
3. **迭代查询**兜底——前两个策略仍然不够时，扩大检索范围再试

最终排名贡献：**实体排序 > 层级扩展 > 迭代查询**。实体排序纠正了最多的语义漂移，大约贡献了 5-6 个百分点的提升。

## 四、完整面试叙述模板（3-4 分钟）

> 这个项目是一个设备手册的智能问答系统。传统 RAG 直接切块做向量检索，准确率大概 70% 左右，核心问题是三个：切块丢失层级上下文、专业术语导致语义漂移、固定召回量无法应对复杂问题。
>
> **预处理阶段**我做了三件事：
> - 第一，PDF 转 Markdown。利用 Markdown 的标题语法天然保留层级结构，然后以标题为边界做语义切块，每个 chunk 自动带上所属章节标签，保证不跨章节、不拆表格。
> - 第二，离线实体提取。用 LLM 批量标注每个 chunk 中的设备名、参数名、数值，存到元数据里。
> - 第三，构建层级向量库。以 parent_title 为关键元数据，text-embedding-v3 做向量化，全部写入 ChromaDB。
>
> **检索阶段**我设计了三个策略：
> - 层级索引解决上下文缺失：命中一个 chunk 后按 parent_title 把同章节 chunks 全拉出来，相当于"命中一点，展开一面"。
> - 实体排序解决语义漂移：query 实时提取实体，和候选 chunks 的预标注实体按 0.5/0.5 权重做混合打分重排序。
> - 迭代查询解决召回不足：LLM 自评答案质量，不够就自动扩大 k 值重查，最多三轮。
>
> 最终检索准确率从约 70% 提升到 80%+。最大的收获是理解了检索策略要和文档结构对齐——从 Markdown 切块保留层级，到检索时利用层级元数据扩展，整条链路是贯通的，不是东拼西凑。

## 七、专题：为什么选 ChromaDB？

这个追问几乎是必问的——面试官想知道你的技术选型是有依据的，而不是"只会用这个"。

### 回答策略：以三个核心理由为主线

选 ChromaDB 归根到底就三条：

> **1. 轻量、便于移植。** ChromaDB 是一个 pip install 就能用的嵌入式库，数据持久化在本地目录，拷贝目录就能备份或迁移到另一台机器，没有任何环境依赖。
>
> **2. 无需引入其他中间件。** 不与应用进程分离，不需要额外起一个服务进程（不像 ES 起 JVM、Milvus 起 etcd+MQ）。整个 RAG 系统就是一个 Python 进程，部署和运维简单得多。
>
> **3. Python 原生，便于开展实验。** 全 Python API，和 PyTorch、LangChain 等生态无缝衔接。做消融实验、调参对比时，不需要连远程服务、不需要切命令行，直接在 notebook 里就能跑通全流程。

### 横向对比（支撑以上三点）

> 选型时也对比了几个主流方案，最终回到这三点上：
>
> - **FAISS**：纯向量检索，性能很强，但不支持 metadata 过滤。我们的层级扩展需要"按 parent_title 拉出同章节 chunks"，FAISS 做不到，得自己在外面再包一层映射关系，维护成本高。同时 FAISS 的 C++ 底层在实验迭代时不如纯 Python 方便。
> - **Milvus**：功能全，支持标量过滤，但需要独立部署 etcd + MQ + 计算节点，引入了一套中间件。设备手册几千页，用 Milvus 属于杀鸡用牛刀，移植到新环境也很麻烦。
> - **Pinecone / Weaviate**：云服务，数据不出内网的安全合规绕不开。而且网络延迟对实验阶段的快速迭代不友好。
> - **ChromaDB**：嵌入式运行、pip install 即用、Python 原生、支持 metadata where 过滤。parent_title 过滤一行代码搞定，数据就是本地一个目录，打包带走就能跑。三点全满足。

### 展现深度（如果面试官继续追问）

> 其实更深层的原因是，ChromaDB 在这个系统里不是瓶颈。检索准确率的瓶颈在切块策略和排序逻辑，不在向量库的 QPS 或延迟。换 FAISS 或 Milvus 不会让 70% 变成 80%，但加上层级索引和实体排序可以。
>
> 选型原则是：**不要在非瓶颈环节过度设计**。如果未来文档量到百万级别、需要分布式检索，再迁到 Milvus 也不晚——向量库的读写接口足够标准，迁移成本很低。

### 面试官可能的连环追问

**追问 1：ChromaDB 性能撑得住吗？**

> 设备手册几千页，chunks 数量在万级别，ChromaDB 完全够。我们的检索延迟主要在 LLM 调用（生成答案），向量检索本身是毫秒级的。实际上 ChromaDB 处理十万级向量都没问题。

**追问 2：如果数据量增长到百万级呢？**

> 十万级以上 ChromaDB 会有性能下降，到时候可以考虑迁移到 Milvus 或 Qdrant。但我们的预处理管线是标准化的——切块+实体标注→写入向量库——换底层存储只需要改写入和查询接口，业务逻辑不动。

**追问 3：有没有考虑过用 Elasticsearch？ES 和 ChromaDB 有什么区别？**

这是面试官考察技术选型深度的关键问题。需要从两个维度回答：**本质区别**和**为什么不选 ES**。

**一、ES 和 ChromaDB 的本质区别：**

| 维度 | Elasticsearch | ChromaDB |
|------|--------------|----------|
| **定位** | 全文检索引擎，后来加了向量能力 | 原生向量数据库 |
| **检索原理** | 倒排索引（BM25）+ 可选的 kNN 向量召回 | 纯向量近似最近邻（ANN）检索 |
| **索引结构** | 分词→倒排表，向量是扩展功能 | HNSW 图索引，向量是一等公民 |
| **部署方式** | Java 服务进程，需要独立部署运维 | 嵌入式 Python 库，随应用进程启动 |
| **资源消耗** | JVM 堆内存 2-4GB 起步 | 进程内内存，百 MB 级别 |
| **强项** | 全文搜索、聚合分析、海量日志/时序 | 语义检索（相似度而非关键词匹配） |
| **过滤能力** | 极其丰富的 DSL 过滤语法 | 基础的 metadata where 过滤 |
| **向量检索成熟度** | v8.0+ 才比较可用，社区相对新 | 原生设计，API 简洁 |

一句话总结：**ES 的核心是"精确匹配关键词"，ChromaDB 的核心是"语义相似度"**。两种检索范式根基不同。

**二、为什么不选 ES？**

> 考虑过 ES，但在这个场景下有三条不选的理由：
>
> **1. 设备手册的查询是"语义问题"而非"关键词搜索"**
>
> 用户问的是"变压器温度过高怎么办"，不是搜"变压器 温度 过高"这几个词。ES 的 BM25 倒排索引对精确词匹配很强，但用户用不同表述问同一个问题——比如换说"油温告警异常怎么处理"——BM25 的召回率会大幅下降。而向量检索天然处理同义改写，这正是我们需要的。
>
> **2. ES 的混合检索（BM25 + kNN）和我们做的事有重叠但不完全相同**
>
> ES 的 Hybrid Search 是 BM25 召回 + kNN 召回 → 融合排序。它的关键词侧是分词匹配，我们的实体排序侧是实体语义匹配。实体匹配的区别在于——"主变压器"和"变压器"在 BM25 里是两个不同的分词，但在实体匹配里能识别出是同一类设备的变体。换句话说，**实体排序是一种比 BM25 更"领域化"的精确匹配**。
>
> **3. 部署成本换不来等比例的收益**
>
> ES 需要独立部署 Java 服务、管理集群、调优 JVM。几千页手册、万级 chunks，用 ES 就是杀鸡用牛刀。而且我们的核心问题不在检索引擎——换 ES 也解决不了层级割裂和实体漂移。ChromaDB 的检索延迟是毫秒级，瓶颈在 LLM 生成答案，检索引擎再快也省不了那部分时间。
>
> 但我也不是说 ES 不好。如果未来需求变成：全文关键词搜索 + 向量语义搜索 + 按设备型号/日期范围过滤的**多条件混合查询**，同时有**多用户高并发**的场景，那 ES 会是更好的选择。技术选型永远是匹配需求，不是哪个更牛用哪个。

**三、面试官可能的进一步追问：**

"那如果用户输入了精确的设备型号比如'X-200型继电器'，用向量反而可能匹配到其他继电器，ES 的倒排索引是不是更好？"

> 这个场景确实存在。我们的处理方式是：实体排序已经覆盖了精确匹配的需求——'X-200型继电器'会被 entity_llm 提取为实体，然后在候选 chunks 中做精确实体匹配，效果等同于关键词精确匹配。所以我们是"向量做宽召回 + 实体做精排"，而不是纯靠向量。
>
> 如果未来精确型号查询的比例很高，可以考虑加一层 ES 倒排索引做前置过滤，再做向量检索——但当前数据量下，万级向量全量检索本身就很快，前置过滤的收益不明显。

**追问 4：ChromaDB 的持久化可靠吗？**

> ChromaDB 底层基于 SQLite，持久化没问题。集合存储在 `persist_directory` 下，直接拷贝目录就能备份和迁移。我们线上跑了一段时间没有丢数据的情况。
